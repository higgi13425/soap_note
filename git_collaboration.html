Collaborating with Git and GitHub: A Beginner’s Guide

Using the Fork, Branch, and Pull Request Workflow

1 Introduction

Welcome! This document will walk you through the basics of collaborating on a project using Git and GitHub. These tools are widely used in data science, software development, and research to track changes and work together on shared projects.

By the end of this guide, you will know how to:

  • Fork a repository to get your own copy
  • Clone the repository to your computer
  • Create a branch to safely make changes
  • Commit and push your changes
  • Open a Pull Request to propose your changes to the original project
What is Git vs. GitHub?

Git is a version control system that runs on your computer. It tracks every change you make to your files over time.

GitHub is a website that hosts your Git repositories online. It also provides tools for collaborating with others.

2 Key Concepts

Before we dive in, here are a few important terms:

Term What It Means
Repository (repo) A folder containing your project files and their complete history
Fork Your own personal copy of someone else’s repository on GitHub
Clone Downloading a copy of a repository to your local computer
Branch A parallel version of the code where you can make changes safely
Commit A saved snapshot of your changes with a descriptive message
Push Uploading your local commits to GitHub
Pull Request (PR) A request for the original project owner to review and merge your changes

3 Step 1: Fork the Repository

A fork creates your own personal copy of someone else’s repository on GitHub. This lets you freely experiment without affecting the original project.

How to fork:

  1. Go to the repository page on GitHub (for example: https://github.com/higgi13425/soap_note)
  2. Click the Fork button in the upper-right corner of the page
  3. GitHub will create a copy of the repository under your own account (e.g., https://github.com/YOUR-USERNAME/soap_note)

You now have your own independent copy of the project!

4 Step 2: Clone Your Fork to Your Computer

Cloning downloads the repository from GitHub to your local machine so you can edit files.

In RStudio, you can do this easily:

  1. Click File → New Project → Version Control → Git
  2. Paste the URL of your fork (not the original) into the “Repository URL” field
  3. Choose a folder on your computer and click Create Project

Or, from the terminal:

git clone https://github.com/YOUR-USERNAME/soap_note.git
cd soap_note
Tip

Always clone your fork, not the original repository. That way your changes stay in your own copy until you’re ready to share them.

5 Step 3: Create a Branch

A branch is like a safe workspace where you can make changes without affecting the main version of the code. It is good practice to always work on a new branch rather than directly on main.

From the terminal, create and switch to a new branch:

git checkout -b my-new-feature

This creates a branch called my-new-feature and switches to it right away. You can name your branch anything descriptive — for example, add-plan-section or fix-typo.

To check which branch you are on:

git branch

The branch with * next to it is your current branch.

Note

In RStudio, you can create and switch branches using the Git pane in the upper-right panel. Click the branch name dropdown and select New Branch.

6 Step 4: Make Your Changes

Now you can open files in the repository and edit them. For example, you might add text to the SOAP note, fix a typo, or add a new section.

After editing a file, save it as you normally would.

7 Step 5: Stage and Commit Your Changes

A commit saves a snapshot of your changes with a message describing what you did. Before committing, you must stage the files you want to include.

From the terminal:

# Stage your changes (add the file to the commit)
git add filename.qmd

# Or stage all changed files at once
git add .

# Commit with a descriptive message
git commit -m "Add assessment section to SOAP note"

In RStudio:

  1. In the Git pane, check the box next to each file you want to stage
  2. Click Commit
  3. Write a short, descriptive commit message
  4. Click Commit again
Writing Good Commit Messages

A good commit message briefly explains what you changed and why.

✅ Good: "Add objective section with vital signs"
❌ Less helpful: "changes" or "stuff"

8 Step 6: Push Your Changes to GitHub

Pushing uploads your local commits to your fork on GitHub.

git push origin my-new-feature

Here, origin refers to your fork on GitHub, and my-new-feature is the name of your branch.

In RStudio, click the green Push arrow in the Git pane.

After pushing, go to your fork on GitHub — you should see your new branch and a prompt to open a Pull Request.

9 Step 7: Open a Pull Request

A Pull Request (PR) is how you propose your changes to the owner of the original repository. It lets them review your changes, leave comments, and ultimately merge your work into the main project.

How to open a Pull Request:

  1. Go to your fork on GitHub
  2. You should see a yellow banner saying “Compare & pull request” — click it
  3. Make sure the base repository is the original (higgi13425/soap_note) and the base branch is main
  4. Make sure the head repository is your fork and the compare branch is my-new-feature
  5. Write a clear title and description explaining what your changes do and why
  6. Click Create Pull Request

The repository owner will be notified and can review your changes, suggest edits, and eventually merge them.

What Happens After You Submit a PR?
  • The owner may merge your PR (your changes are added to the main project 🎉)
  • The owner may request changes (they ask you to fix or improve something before merging)
  • The owner may close the PR (your changes are not accepted this time)

If changes are requested, simply make the edits on the same branch, commit, and push — the PR will update automatically.

10 The Full Workflow at a Glance

        │
        │  Fork
        ▼

        │
        │  Clone

        ▼
  Your Branch (my-new-feature)

        │  Pull Request
        ▼
Original Repo (review & merge)

11 Summary

Here is a quick recap of each step:

  1. Fork the repository on GitHub to create your own copy
  2. Clone your fork to your local computer
  3. Create a branch to isolate your changes
  4. Edit the files you want to change
  5. Stage and commit your changes with a descriptive message
  6. Push your branch back to your fork on GitHub
  7. Open a Pull Request to propose your changes to the original repository

That’s it! With practice, this workflow will become second nature. It is the same process used by millions of developers and data scientists around the world to collaborate on projects of all sizes.
